4.2 ALIGNMENT WITH NOTION REVIEW SYSTEM
------------------------------------------
  ISSUE - The admin page and the Notion review system are disconnected.
  There is no link or reference from the admin page to the Notion review
  status of a problem. The admin page shows Supabase's `status` field
  (needs_review, ready, sent) but does not show whether the problem has
  a Notion page, what the Notion review status is, or provide a link to
  open the Notion review page.

  RECOMMENDATION: Add a "Notion" column to the admin table that shows:
    - An icon/badge indicating Notion sync status
    - A clickable link to open the Notion review page directly
    - The notion_page_id is already stored in Supabase, so constructing
      the URL (https://notion.so/{page_id}) is straightforward

  ISSUE - The admin page uses English column headers (Problem ID, Year,
  Exam, No., Status, Actions) but the Notion database uses Korean

  inconsistency can confuse operators switching between the two

  RECOMMENDATION: Either standardize on Korean for both interfaces
  (recommended, since the target audience is Korean), or provide a
  language toggle.

  ISSUE - The admin page filter labels use a mix of Korean and English.
  "ê²€ìƒ‰" (search), "ì—°ë„" (year), "ì‹œí—˜" (exam), "ë°°ì " (score), "ìƒíƒœ"

5. OVERALL SYSTEM COHESION
================================================================================

5.1 DATA FLOW SUMMARY
------------------------
  Supabase (source of truth) -> sync_to_notion.py -> Notion (review UI)
  Notion (reviewed) -> sync_to_supabase method -> Supabase (updated)
  Supabase (ready) -> Admin page -> KakaoTalk (delivery)

  This three-stage pipeline is architecturally sound. Each system plays
  to its strengths: Supabase for structured data, Notion for human-
  friendly review, Admin page for operational actions.


5.2 KEY GAPS IN THE FLOW
---------------------------
  (a) NO BIDIRECTIONAL SYNC AUTOMATION: The Notion -> Supabase sync
      must be manually triggered. Consider a scheduled job or webhook
      to automate this.

  (b) NO NOTIFICATION ON REVIEW COMPLETION: When a reviewer marks a
      problem as "ê²€ìˆ˜ ì™„ë£Œ" in Notion, there is no automated notification
      to the operator that problems are ready for Supabase sync. Consider
      integrating Notion webhook or polling.

  (c) NO REVIEW PROGRESS DASHBOARD: Neither the Notion database views
      nor the admin page provide an aggregate view of review progress
      (e.g., "45/60 problems reviewed for 2026 CSAT"). The admin page
      has total counts but not per-year or per-exam breakdowns that
      would show review completion percentage.

  (d) HINT EDITING FRICTION: Hints are stored in a separate `hints`
      table in Supabase and displayed as Notion page body content (in
      toggles). But they are also stored as Notion properties (íŒíŠ¸1,
      íŒíŠ¸2, íŒíŠ¸3). If a reviewer edits a hint in the property field
      vs. the page body toggle, it is unclear which takes precedence
      during reverse sync. The sync_to_supabase method reads from
      properties, not from page body blocks. This should be documented
      or the page body hints should be read-only.

-------------------------------------
  Current reviewer workflow:
    1. Open Notion database
    2. Filter by ìƒíƒœ = "ê²€ìˆ˜ í•„ìš”"
    3. Open a problem page

  FRICTION POINTS:
    - Step 4-5: Scrolling up and down between image and answer info
    - Step 9: Must change property manually (no button or shortcut)
    - No visual indicator of completion progress within a review session
    - No "next problem" navigation from within the page
